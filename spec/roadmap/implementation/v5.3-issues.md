## v5.3 — Three-level cascade and the pyramid

### NADI-049 — `demos/vyaghra.py` three-species scenario

**Description:**
The three-species demo: Bindu + Chara + Vyaghra on the Tejas field, wired over the same World/demo-loop pattern as `demos/bindu.py` (v1.3) and `demos/chara.py` (v4.3). A world config carries every tunable; the seed is explicit on the CLI. (`demos/vyaghra.py` + a world config.)

**What needs to be done:**
- `demos/vyaghra.py`: spawn all three species from their descriptors (`bindu.yaml`, `chara.yaml`, `vyaghra.yaml`) at seeded positions on the Tejas field; step the world N ticks; render and log counters each tick.
- A world config for the scenario — grid size, seed, Tejas patch parameters, `MAX_POP`, per-species initial counts, demo tick count — one config per demo, passed into the World; nothing tunable lives in code (ARCHITECTURE §Configuration and parameters).
- A `--seed` CLI flag overriding the config seed — the seed is always explicit, in the config or on the CLI, never implicit.
- The renderer shows all three species plus the hunt/flee states from v5.2, with the per-level population header from NADI-046.
- Follow the established demo-runner conventions (tick count/pacing flags as in prior demos) so the scenario slots straight into the smoke harness for NADI-050.

**Dependencies:** NADI-048

**Expected result:**
One command runs the full three-level world — light, producers, herbivores, predators, hunts and flights — reproducibly under a chosen seed.

**Acceptance criteria:**
- [ ] `demos/vyaghra.py` runs Bindu + Chara + Vyaghra on Tejas, instantiated from their YAML descriptors
- [ ] All scenario parameters live in the world config + species YAML; none in code
- [ ] `--seed` sets the run's seed; two runs at the same seed produce byte-identical counter logs
- [ ] The renderer shows the three species (and hunt/flee states) with per-level populations in the header
- [ ] The demo runs K ticks without error under the default config (pre-tuning sanity)

### NADI-050 — Cascade tuning + the pyramid smoke guard

**Description:**
Tune the three-level world into coexistence and encode the v5 readiness criterion as a regression guard: a **long fixed-seed smoke run** in which all three populations stay above zero, oscillate without exploding, and hold the abundance pyramid (Bindu > Chara > Vyaghra) — a collapse of any level fails the guard. Tuning touches the world config and the species YAML **only** (ARCHITECTURE §Configuration and parameters); an engine change needed for balance is an abstraction leak to be raised, not patched in. (config + `species/*.yaml` + the smoke test.)

**What needs to be done:**
- Tune for three-level coexistence: Tejas patch supply, per-species initial counts, the ~10% `gain`/`yields` transfer per level, reproduction thresholds/cooldowns/caps and costs, lifespans, hunt radius — all in the two configuration homes, nothing in engine code.
- Steer the tuning with the per-level time-series from NADI-046: producers most numerous, predators fewest, out-of-phase oscillation across levels rather than flatline or runaway (the v4.3 two-level picture extended one level up).
- The smoke guard — a long fixed-seed run of K ticks under the tuned reference parameters asserting: (a) all three populations stay above zero for the whole run; (b) populations stay inside configured bands (no explosion to `MAX_POP`, no extinction) — stable-if-oscillating; (c) the abundance-pyramid ordering holds over the run (asserted on windowed means so ordinary oscillation cannot false-fail; the tolerance defined explicitly in the test).
- Wire the guard into CI as the v5 smoke test — pure Python, no network, no Node (ARCHITECTURE §Testing and CI); it is the regression net on the tuned parameters: any future change that collapses a level fails it.
- Check in the tuned reference parameters as the scenario's config; the demo and the guard share one source of truth.
- Determinism harness: two runs at the guard's seed produce byte-identical counter logs.

**Dependencies:** NADI-049, NADI-046

**Expected result:**
A standing three-level ecosystem — stable-if-oscillating coexistence with the pyramid intact — protected by a CI smoke guard (Layer 6 readiness).

**Acceptance criteria:**
- [ ] Tuning is confined to the world config + species YAML (no engine changes)
- [ ] Smoke: a long fixed-seed run keeps all three populations above zero for the whole run
- [ ] Smoke: populations stay inside configured bands — stable-if-oscillating, no explosion, no extinction
- [ ] Smoke: the abundance-pyramid ordering (Bindu > Chara > Vyaghra) holds over the run; a collapse of any level fails the guard
- [ ] The guard runs in CI on the checked-in reference config; two same-seed runs produce byte-identical counter logs

## v5.3 scope notes

**Total effort:** ~1 week for a single developer (NADI-049 is thin scaffolding; the bulk is the tuning loop inside NADI-050).

**Critical path:** NADI-049 → NADI-050 (the scenario, then the balance that proves the version).

**Phase DoD (ROADMAP §v5.3, Layer 6 readiness):** stable-if-oscillating coexistence of all three levels over a long smoke run; the abundance pyramid holds (Bindu > Chara > Vyaghra).

**Tuning note:** "Balance is hardest at v4–v5. The food chain tends to break under random movement (prey and predator rarely meet)" (ROADMAP §Notes on the ladder) — the directed movement from v2/v4 and the hunt/flee signalling from v5.2 are exactly the survival tools that make this pyramid stand. If coexistence cannot be reached from config and YAML alone, that is a design finding to surface, not a licence to touch the engine.
