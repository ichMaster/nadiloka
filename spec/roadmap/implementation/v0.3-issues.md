## v0.3 — ASCII renderer and the light demo

### NADI-009 — ASCII renderer

**Description:**
The first renderer (ARCHITECTURE §Observation layer): `render.py` draws the grid in the terminal, mapping cell intensity to brightness glyphs, under a header line of live counters (tick, total energy, live patches). Strictly read-only over the world. (`nadiloka/render.py`.)

**What needs to be done:**
- An intensity → brightness glyph map (a dark-to-bright ramp); the mapping must be **total**: every value in `0..FOOD_MAX` — including 0, `FOOD_MAX`, and everything between — maps to a glyph; no input can raise.
- Render the W×H grid one glyph per cell, in a single pass, with no world mutation.
- A header line showing the live counter subset: tick, total field energy, live patch count (the counters from NADI-008).
- Keep the renderer loop-independent: it takes a world (or its field + counters) and produces one frame — `demos/light.py` (NADI-010) owns the loop.
- Unit test: the intensity → glyph mapping is total across the full range, boundary values included.

**Dependencies:** NADI-006

**Expected result:**
Any field state renders as an ASCII frame with a correct counter header, mutating nothing.

**Acceptance criteria:**
- [ ] Intensity → glyph mapping is total over `0..FOOD_MAX` (unit test, boundaries included)
- [ ] The grid renders one glyph per cell at any W×H from config
- [ ] The header line shows tick, total energy, live patches
- [ ] Rendering never mutates the world (read-only observation)
- [ ] The renderer is loop-independent (callable per frame)

### NADI-010 — `demos/light.py` scenario

**Description:**
The v0 demo: wire World + Tejas + renderer over a loop — build a world from the reference config, step it, render each tick. The seed comes from the CLI (`--seed`), never implicitly (ARCHITECTURE §Configuration and parameters). (`demos/light.py`.)

**What needs to be done:**
- `demos/light.py`: construct the `World` from a per-demo config (grid size, patch parameters, tick count), loop `World.step()`, render each tick via NADI-009's renderer.
- `--seed` on the CLI (plus tick-count/config overrides as needed); the explicit seed feeds the World's single RNG.
- Reasonable reference parameters checked in with the demo config — the same parameters the smoke run (NADI-011) pins.
- Keep the counter log lines flowing (NADI-004/NADI-008) alongside the rendered frames, so the run stays measurable as well as visible.
- Exercise the read-only invariant: running with rendering on or off under the same seed produces identical counter logs (formalized in NADI-011's suite).

**Dependencies:** NADI-009, NADI-007

**Expected result:**
`python demos/light.py --seed 42` shows a living, moving light field in the terminal: patches ignite, glow, fade, and reappear.

**Acceptance criteria:**
- [ ] `demos/light.py` wires World + Tejas + renderer over a loop
- [ ] `--seed` (explicit) on the CLI; the run is reproducible under it
- [ ] Reference parameters live in the demo config, not in code
- [ ] Counter lines are emitted alongside the rendered frames
- [ ] Same seed, renderer on vs off → identical counter logs

### NADI-011 — Layer smoke run: the living light field

**Description:**
The readiness gate for Layer 1 (v0): a fixed-seed smoke test of K ticks at the reference parameters, asserting the energy band from NADI-008 and that every frame renders without error — the ROADMAP §v0.3 DoD encoded as a test (ARCHITECTURE §Testing and CI: each version's readiness criterion is its smoke test).

**What needs to be done:**
- A smoke test running `demos/light.py`'s scenario (or its extracted loop) for K ticks under a fixed seed and the reference config.
- Assert the energy band: total field energy stays within the configured band for the whole run (reusing NADI-008's band and parameters).
- Assert rendering: every tick's frame renders without raising (frames may go to a null sink in CI).
- Assert determinism end to end: two smoke runs with the same seed produce byte-identical counter logs.
- Keep the smoke test in the standard `pytest` suite — pure Python: no network, no Node, no pygame (ARCHITECTURE §Testing and CI).

**Dependencies:** NADI-010

**Expected result:**
The smoke suite proves v0's readiness: a living, moving light field whose total energy holds steadily at the intended level.

**Acceptance criteria:**
- [ ] Smoke test: fixed-seed run of K ticks at the reference parameters
- [ ] Total energy stays within the configured band over the run
- [ ] Every frame renders without error
- [ ] Two same-seed smoke runs produce byte-identical counter logs
- [ ] The smoke test runs in plain `pytest` CI (no network, no Node, no pygame)
