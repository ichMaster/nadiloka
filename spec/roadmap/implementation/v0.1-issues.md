# v0.1 — GitHub Issues

Issues for phase **v0.1 — Skeleton and the Meru tick** (version **v0 — World and light (Tejas)**), derived from the per-phase Tasks list in [ROADMAP.md](../../ROADMAP.md) (§v0.1). This file is scoped to a single phase; v0.2–v0.4 get their own breakdowns. Each issue ships with the tests that encode its DoD (see [ARCHITECTURE.md](../../ARCHITECTURE.md) §Testing and CI).

v0.1 is the **foundation**: an empty Nadiloka that ticks deterministically. It stands up the `nadiloka/` engine package, the **`World`** container (config, one seeded `random.Random`, a tick counter, an empty population), and **`Meru.step()`** with the fixed five-phase order stubbed — update Tejas → rebuild grid → act → flush Nadi → death/cleanup — each phase a no-op hook. The point is to lock the phase order and the reproducibility seam before any state exists; every later version fills these hooks and never reorders them. NADI IDs start here (NADI-001…).

### NADI-002 — `World` container: config, seeded RNG, tick counter, population

**Description:**
The `World` object that holds everything (ARCHITECTURE §Component overview): the config, **one** seeded `random.Random`, the tick counter, and the (still empty) population dict. This issue fixes the **RNG discipline** invariant — the reproducibility seam of the whole project. (`nadiloka/world.py`.)

**What needs to be done:**
- `World` holding: a config object, a single `random.Random(seed)`, an integer tick counter, and an empty population dict (`id → Digitant`, populated from v1).
- A world-config dataclass per ARCHITECTURE §Configuration and parameters — the seed is always explicit (config or CLI), never implicit; grid/Tejas parameters are placeholders filled in v0.2.
- **RNG discipline:** the one `Random` instance is owned by the World and passed explicitly to whatever needs randomness; module-level `random.*` calls are forbidden — they silently break reproducibility (ARCHITECTURE §Key invariants and pitfalls).
- Nothing tunable lives as an engine constant: every number belongs to the config.
- Unit test: two `World`s built with the same seed yield an identical RNG draw sequence; different seeds diverge.

**Dependencies:** NADI-001

**Expected result:**
A `World` constructs from a config with an explicit seed; its RNG draws reproduce exactly under the same seed.

**Acceptance criteria:**
- [ ] `World` holds config, one seeded `random.Random`, a tick counter, and an empty population dict
- [ ] The seed is explicit in the config (CLI wiring lands with the runner, NADI-004); no implicit seeding
- [ ] No module-level `random.*` anywhere in the engine; the World's RNG is passed explicitly
- [ ] Tunables live in the config, not as engine constants
- [ ] Unit test: same seed → identical RNG draw sequence; different seed → different sequence

### NADI-003 — `Meru.step`: the five-phase hook order

**Description:**
One turn of Meru — `World.step()` — calling the five phases in fixed order (ARCHITECTURE §One tick cycle): **update Tejas → rebuild the spatial grid → digitants act → flush Nadi → death and cleanup**. Each phase is a no-op hook here; later versions fill them (Tejas in v0.2, act/death in v1, flush in v3) and **never reorder them**. (`nadiloka/world.py`.)

**What needs to be done:**
- `World.step()` (the Meru turn): call the five phase hooks in the fixed order, then advance the tick counter — single and synchronous for all.
- Each phase an explicit, named no-op hook (e.g. `_update_tejas`, `_rebuild_grid`, `_act`, `_flush_nadi`, `_cleanup`) — the seams the roadmap fills version by version.
- Document the deterministic phase order as an invariant at the hook site (ARCHITECTURE §Key invariants and pitfalls).
- Unit test: a recording spy proves the five phases are called once each per tick, in the fixed order.
- Unit test: N steps on an empty world increment the tick counter to N with no state and no error.

**Dependencies:** NADI-002

**Expected result:**
An empty world steps N times, phases firing once each in the fixed order every tick.

**Acceptance criteria:**
- [ ] `World.step()` calls exactly five phase hooks in the order update Tejas → rebuild grid → act → flush Nadi → death/cleanup
- [ ] Each phase is a named no-op hook (the extension seam for v0.2/v1/v3)
- [ ] The tick counter increments once per step
- [ ] Unit test: the five phases fire once each per tick, in order (spy/recording test)
- [ ] Unit test: N steps on an empty world run with no error; tick count = N

### NADI-004 — Demo runner + counters scaffold

**Description:**
The first runnable artifact and the measurable half of observability: a `demos/` runner stepping the world N ticks, and the counters scaffold emitting structured log lines (tick, population, energy) that every readiness criterion from here on is asserted against (ARCHITECTURE §Counters and measurement). Delivers the v0.1 DoD.

**What needs to be done:**
- A `demos/` runner: build a `World` from a config, step it N ticks; N and `--seed` on the CLI (the seed is always explicit — ARCHITECTURE §Configuration and parameters).
- Counters scaffold: per tick, emit a structured log line with `tick`, `population` (0 for now), `energy` (0.0 for now) — the format tests and demos consume; the v0 set (total Tejas energy, live patches) fills in with v0.2.
- Counter emission is strictly read-only over the world (ARCHITECTURE §Observation layer).
- **Determinism harness** (ARCHITECTURE §Testing and CI): a test running the same seed + config twice and asserting byte-identical counter logs — the cheapest regression net; it runs on every CI pass from now on.

**Dependencies:** NADI-003

**Expected result:**
`World.step()` runs N times with no state and no error; two runs with the same seed produce identical counter logs (the v0.1 DoD verbatim).

**Acceptance criteria:**
- [ ] A `demos/` runner steps a fresh world N ticks from a config; `--seed` on the CLI
- [ ] Each tick emits a structured counter line (tick, population, energy)
- [ ] Counters never mutate the world (read-only observation)
- [ ] Unit test: same seed + config, two runs → byte-identical counter logs
- [ ] Unit test: N ticks with no state complete without error
