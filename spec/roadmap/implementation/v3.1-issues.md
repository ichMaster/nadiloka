# v3.1 — GitHub Issues

Issues for phase **v3.1 — The Nadi bus** (version **v3 — Nadi and communication**), derived from the per-phase Tasks list in [ROADMAP.md](../../ROADMAP.md) (§v3.1). This file is scoped to a single phase; v3.2 gets its own breakdown. Each issue ships with the tests that encode its DoD (see [ARCHITECTURE.md](../../ARCHITECTURE.md) §Testing and CI).

v3.1 is the **bus**: agents stop being solitary. `nadi.py` gets the `Message` model and the deferred queue — `emit()` only queues during a tick, `flush()` delivers the whole queue in the flush-Nadi phase, and replies emitted during delivery wait for the next tick, so cascades stay bounded and the tick stays deterministic ([ARCHITECTURE.md](../../ARCHITECTURE.md) §Event bus (Nadi)). On top of the queue come three addressing modes with deterministic (id-sorted) recipient resolution, and the subscription registry that turns a delivered message into a pysm event on the recipient's brain (payload in `cargo`, never `input=`). No behaviour changes yet — producers start signalling in v3.2.

## v3.1 — The Nadi bus

### NADI-030 — Message model + deferred queue

**Description:**
The `Message` model and the deferred queue at the heart of Nadi. `emit()` during a tick only queues; `flush()` delivers the whole queue in the flush-Nadi phase (phase 4 of `Meru.step()`, fixed since NADI-003); replies emitted *during* delivery go out the **next** tick. This is the mechanism that keeps cascades bounded and the tick deterministic ([ARCHITECTURE.md](../../ARCHITECTURE.md) §Event bus (Nadi)). (`nadiloka/nadi.py`.)

**What needs to be done:**
- `nadi.py`: `Message{topic, sender, payload, addressing}` per ARCHITECTURE §Event bus (Nadi).
- `emit(message)` appends to the pending queue and returns — nothing is ever delivered mid-tick.
- `flush()` delivers the whole queue in the flush-Nadi phase; anything emitted *while* flushing (a reply) lands in the next tick's queue, never the one being drained.
- FIFO delivery: `flush()` delivers strictly in emit order (ARCHITECTURE §Key invariants — Deterministic Nadi order).
- Wire `flush()` into the flush-Nadi hook of `Meru.step()`; the other four phases stay untouched.
- Unit tests per ROADMAP §v3.1 Tests: emit-during-tick is delivered at that tick's flush; a reply emitted during delivery is delivered one tick later; FIFO order holds across mixed emitters.

**Dependencies:** NADI-003

**Expected result:**
A message emitted this tick is delivered in the same tick's flush; a reply is delivered the next tick; delivery order is FIFO by emit order.

**Acceptance criteria:**
- [ ] `Message{topic, sender, payload, addressing}` defined in `nadi.py`
- [ ] `emit()` only queues during a tick; no delivery happens outside `flush()`
- [ ] `flush()` runs in the flush-Nadi phase of `Meru.step()` and drains the whole queue
- [ ] Unit test: a reply emitted during delivery goes out the **next** tick (one bounded hop per tick)
- [ ] Unit test: delivery is FIFO by emit order

### NADI-031 — Addressing modes with deterministic resolution

**Description:**
The three addressing modes — `target=id` (one specific agent), `radius=r` (neighbours via the spatial hash), otherwise global — with recipients resolved in a **stable id-sorted order**, so delivery never depends on hash or dict ordering ([ARCHITECTURE.md](../../ARCHITECTURE.md) §Event bus (Nadi), §Key invariants). (`nadiloka/nadi.py`.)

**What needs to be done:**
- `target=id`: resolve to exactly one live agent; a dead or absent target receives nothing (the message is dropped, counted, never crashes the flush).
- `radius=r`: resolve neighbours within radius `r` of the sender via the spatial hash (NADI-019; rebuilt each tick in the grid phase, so flush sees this tick's positions).
- No `target`/`radius` → global: all live agents.
- Stable ordering: for radius and global addressing, recipients are resolved **sorted by agent id** — never hash/dict iteration order (the Deterministic Nadi order invariant).
- Unit tests per ROADMAP §v3.1 Tests: each mode targets the right set; radius hits **exactly** the in-range set (in-range agents included, out-of-range excluded, boundary cases pinned); recipient order is id-sorted regardless of population-dict insertion order.
- Determinism harness check: two same-seed runs produce identical delivery logs (ARCHITECTURE §Testing and CI).

**Dependencies:** NADI-030, NADI-019

**Expected result:**
Radius addressing hits exactly the in-range neighbours; targeted and global messages reach the right sets; delivery order is deterministic under any population ordering.

**Acceptance criteria:**
- [ ] `target=id` delivers to exactly that agent (and safely to no one if it is gone)
- [ ] Unit test: `radius=r` resolves via the spatial hash and hits exactly the in-range set
- [ ] Messages with no target/radius deliver globally to all live agents
- [ ] Unit test: recipients resolve in stable id-sorted order — never hash/dict order
- [ ] Two same-seed runs produce identical delivery order (determinism harness)

### NADI-032 — Subscription registry + brain wiring + counters

**Description:**
The subscription registry keyed on the Jati `signals` field, delivery into the recipient's pysm brain, and per-topic counters. A species declares what it emits and listens to declaratively; on delivery a message becomes a pysm event on the recipient's brain with the payload passed via `cargo` — **never** `input=`, the pysm pitfall that silently drops events ([ARCHITECTURE.md](../../ARCHITECTURE.md) §Event bus (Nadi), §Key invariants). This is the seam v3.2 producer signals and v5.2 hunt/flee ride on. (`nadiloka/nadi.py`, `nadiloka/jati.py`, `nadiloka/digitant.py`.)

**What needs to be done:**
- Extend the Jati loader: species declare `signals` (emits / listens topic lists) in the descriptor (ARCHITECTURE §Species descriptive language); validate against known topics.
- A subscription registry mapping topic → listening agents, maintained as digitants are born and die.
- Delivery filter: only agents whose Jati lists the topic under listens receive it — even under radius or global addressing.
- On delivery, dispatch a pysm event on the recipient's brain (NADI-017) with the message payload in `cargo` (never `input=`); a pinning test for the cargo pitfall per ARCHITECTURE §Testing and CI.
- Per-topic counters: emitted and delivered counts (ARCHITECTURE §Counters and measurement, v3 row), exposed through the counters scaffold as structured log lines.
- Unit tests: subscriptions honour the Jati `signals` list (a non-listener in range receives nothing); counters match the actual emitted/delivered totals.

**Dependencies:** NADI-031, NADI-017

**Expected result:**
A species subscribes by descriptor alone; a delivered message fires a pysm event with its payload in `cargo` on each listening recipient; the counters report per-topic traffic.

**Acceptance criteria:**
- [ ] Jati `signals` (emits/listens) loads, validates, and drives the subscription registry
- [ ] Only listeners receive a topic, under every addressing mode
- [ ] Delivery fires a pysm event with the payload in `cargo`; a pinning test proves `input=` is never used
- [ ] Per-topic emitted/delivered counters match the traffic (unit test)
- [ ] Determinism harness: same seed → identical per-topic counter logs across two runs
