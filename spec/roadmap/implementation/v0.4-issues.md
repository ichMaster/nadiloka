# v0.4 — GitHub Issues

Issues for phase **v0.4 — Web visualization (React)** (version **v0 — World and light (Tejas)**), derived from the per-phase Tasks list in [ROADMAP.md](../../ROADMAP.md) (§v0.4). This file is scoped to a single phase; v0.1–v0.3 have their own breakdowns. Each issue ships with the tests that encode its DoD (see [ARCHITECTURE.md](../../ARCHITECTURE.md) §Testing and CI).

v0.4 adds the **web observation layer** — a second renderer alongside the ASCII one, never a second engine. The Python side serializes read-only **world-state snapshots** per tick, a small server (FastAPI + uvicorn, **optional deps** — the engine never imports them) streams them over WebSocket, and a **React (Vite) app** in `web/` draws the Tejas field on a canvas; digitants, species colours, and Nadi activity ride the same channel in later versions, extending the payload, not the protocol (ARCHITECTURE §Observation layer). The invariant is absolute: strictly observation, never input into the tick — a same-seed run with the web layer on or off produces identical counter logs, and without `--web` (or Node) everything from v0.3 works unchanged.

## v0.4 — Web visualization (React)

### NADI-012 — JSON snapshot schema + serializer

**Description:**
The engine-boundary contract (ARCHITECTURE §Observation layer): a **JSON world snapshot** serialized each tick — `{tick, grid: {w, h}, tejas: [nonzero cells], agents: [], counters}` — **versioned and additive**, so later versions (digitants in v1, species colours, Nadi activity) extend the payload, not the protocol. (`nadiloka/snapshot.py`.)

**What needs to be done:**
- The schema: `tick`; `grid: {w, h}`; `tejas` as a list of **nonzero** cells (e.g. `[x, y, intensity]`) — empty cells omitted, so payload size tracks live light, not grid area; `agents: []` (empty until v1, present from day one so the shape never changes); `counters` (tick, total energy, live patches).
- A schema-version field and the additive rule: consumers must ignore unknown fields, so new payload keys never break an old viewer.
- The serializer reads the world and builds the snapshot — pure read, no mutation, cheap enough to run every tick.
- Fixture snapshots checked in for the frontend render test (NADI-014) and any other consumer test.
- Unit tests: faithful round-trip (field cells and counters survive serialize → parse); empty cells omitted; a snapshot carrying extra/unknown fields still parses (additivity).

**Dependencies:** NADI-007

**Expected result:**
Every tick can be captured as a small, versioned JSON snapshot that round-trips faithfully and is safe to extend.

**Acceptance criteria:**
- [ ] Snapshot schema `{tick, grid: {w, h}, tejas: [nonzero cells], agents: [], counters}` with a version field
- [ ] The serializer is read-only over the world and runs per tick
- [ ] Unit test: field cells and counters round-trip faithfully
- [ ] Unit test: empty (zero-intensity) cells are omitted
- [ ] Unit test: unknown/additive fields do not break parsing; fixture snapshots checked in

### NADI-013 — Read-only snapshot server

**Description:**
The transport (`nadiloka/server.py`): a small read-only server on **FastAPI + uvicorn as optional deps** — the engine never imports them (ARCHITECTURE §Stack and dependencies) — that drives the demo loop, streams one snapshot per tick over WebSocket, and serves the built React bundle. Reached via a `--web` flag on the demo runner.

**What needs to be done:**
- `server.py` imports FastAPI/uvicorn locally; declare them as an optional extra (e.g. `nadiloka[web]`); nothing on the engine's import path touches them — v0.3 must run with them absent.
- Drive the demo loop: the server owns pacing, calls `World.step()`, serializes via NADI-012, and pushes one snapshot per tick to connected WebSocket clients.
- Serve the built React bundle (`web/dist`) as static files; a clear message when the bundle is not built.
- `--web` on the demo runner (`demos/light.py`): with the flag, the loop runs under the server; without it, the v0.3 terminal path runs exactly as before; `--web` with the extras missing fails with a clear install hint, not a traceback.
- Tick-rate/pace control: a configurable ticks-per-second cap plus the pause/speed commands from the viewer — **pacing only**; no control message ever reaches world state.
- Unit test (runs without the extras installed): the `--web`-off path never imports FastAPI/uvicorn.

**Dependencies:** NADI-012

**Expected result:**
`python demos/light.py --web` serves the viewer and streams the live field, one snapshot per tick; without the flag or the extras, nothing changes.

**Acceptance criteria:**
- [ ] FastAPI + uvicorn are optional deps; the engine and the `--web`-off path never import them
- [ ] One snapshot per tick streams over WebSocket to connected clients
- [ ] The built React bundle is served; missing-bundle and missing-dep cases fail with clear messages
- [ ] `--web` flag on the demo runner; without it the v0.3 behaviour is unchanged
- [ ] Tick-rate/pace control paces the loop only — no control message mutates world state

### NADI-014 — React viewer app

**Description:**
The browser renderer (`web/`, Vite + React): a canvas grid drawing cell intensity as brightness, a counter header matching the ASCII renderer, and play/pause/speed controls that pace the demo loop and never mutate world state. Frontend tests run against fixture snapshots — no live server in CI (ARCHITECTURE §Testing and CI).

**What needs to be done:**
- Scaffold the Vite app in `web/`; connect to the server's WebSocket and parse snapshots per the NADI-012 schema, ignoring unknown fields (the additive rule).
- Canvas rendering: `grid: {w, h}` sizes the canvas; each `tejas` cell draws with brightness proportional to intensity; omitted cells stay dark — the browser twin of the ASCII glyph ramp.
- A header of counters from the snapshot: tick, total energy, live patches — the same numbers the ASCII header shows.
- Play/pause and speed controls sending pacing commands to the server (NADI-013); they pace the loop, never mutate state; the UI reflects the paused/running state.
- A minimal render test on a checked-in fixture snapshot: the expected cells and counters appear — run by `web/` tooling (e.g. Vitest), with **no live server and no Python** in the loop; engine CI stays Node-free.

**Dependencies:** NADI-012

**Expected result:**
The browser shows the living, moving Tejas field, updating each tick, with counters matching the ASCII renderer.

**Acceptance criteria:**
- [ ] Vite React app in `web/` connects and parses snapshots per the schema (unknown fields ignored)
- [ ] Canvas renders intensity as brightness on a w×h grid; empty cells stay dark
- [ ] Counter header shows tick, total energy, live patches (matching the ASCII header)
- [ ] Play/pause/speed send pacing commands only — no state-mutation path in the client
- [ ] Fixture-snapshot render test passes in `web/` with no live server (engine CI needs no Node)

### NADI-015 — Read-only guard and determinism proof

**Description:**
The invariant that makes the web layer safe (ARCHITECTURE §Key invariants and pitfalls: observation is read-only): prove there is **no write path from viewer to world**, and that the whole v0.4 stack changes nothing about a run — the determinism half of the v0.4 DoD.

**What needs to be done:**
- Audit and pin the boundary: the only upstream messages are pacing controls, handled entirely by the loop driver; no handler reaches `World` state; the serializer stays pure-read (pinned by a test at the interface seam).
- The determinism proof: a same-seed run with the web layer **on vs off** produces **byte-identical counter logs** (the ARCHITECTURE §Testing and CI determinism harness, now spanning the web path).
- Pause/speed test: pausing and speed changes mid-run affect wall-clock pacing only; the counter log for N ticks is identical to an unpaced run's.
- The fallback guarantee: without `--web` — or on a machine with no Node and no web extras installed — everything from v0.3 (demo, renderer, smoke suite) works unchanged; pin it with a test environment or CI job that lacks the extras.
- Fold these guards into the standing regression net: they run on every CI pass, like the v0.1 determinism harness.

**Dependencies:** NADI-013, NADI-014

**Expected result:**
The web layer is provably pure observation: toggling it (or lacking its dependencies) never changes a run.

**Acceptance criteria:**
- [ ] No write path from viewer to world (pacing-only boundary pinned by test)
- [ ] Same seed, web on vs off → byte-identical counter logs
- [ ] Pause/speed changes affect pacing only; the N-tick counter log is unchanged
- [ ] Without `--web` (or without Node/web extras) the full v0.3 behaviour and test suite pass unchanged
- [ ] The guards join the per-pass CI regression net
