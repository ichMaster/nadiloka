# v0.2 — GitHub Issues

Issues for phase **v0.2 — Space and the Tejas field** (version **v0 — World and light (Tejas)**), derived from the per-phase Tasks list in [ROADMAP.md](../../ROADMAP.md) (§v0.2). This file is scoped to a single phase; v0.1, v0.3, and v0.4 have their own breakdowns. v0.2 gives the empty world **space and light**: the W×H bounded grid (coords `(x, y)`, no edge wrapping) and the **Tejas** scalar field driven from the update-Tejas phase stubbed in v0.1. Patches ignite at seeded random cells, illuminate their radius, decay over a lifetime, disappear, and respawn elsewhere, holding a target number of live patches — the resource dynamics every trophic level later feeds on. The phase closes with the total-field-energy counter and the energy-band test that becomes the standing "no runaway, no collapse" guard.

## v0.2 — Space and the Tejas field

### NADI-005 — Grid and neighbourhood helpers

**Description:**
The spatial scaffold (ARCHITECTURE §World model): a W×H **bounded** grid, coordinates `(x, y)`, **no edge wrapping**, with the neighbourhood helpers the Tejas patches use now and the spatial hash/digitants use from v1. (`nadiloka/world.py` or a dedicated grid module.)

**What needs to be done:**
- A W×H grid with `(x, y)` coordinates; W and H come from the world config, never engine constants (ARCHITECTURE §Configuration and parameters).
- Bounded, no wrapping: explicit in-bounds checks; out-of-bounds access is impossible or an explicit error, never a silent wrap.
- Neighbourhood helpers: the cells within radius r of a center, clipped at the edges (a query near a border returns a truncated disc).
- Iteration/query helpers with a **stable, deterministic order** — never dict/hash order (ARCHITECTURE §Key invariants and pitfalls, deterministic order).
- Unit tests: bounds behaviour at corners and edges; neighbourhood-of-radius correctness including edge clipping.

**Dependencies:** NADI-002

**Expected result:**
A bounded grid the Tejas field (and later the spatial hash) sits on, with correct, deterministic neighbourhood queries.

**Acceptance criteria:**
- [ ] W×H bounded grid with `(x, y)` coordinates; W, H from the world config
- [ ] No edge wrapping; out-of-bounds is explicit, never silent
- [ ] Neighbourhood queries return the cells within radius r, clipped at the edges
- [ ] Iteration/query order is stable and deterministic (no hash/dict order)
- [ ] Unit tests cover corner/edge neighbourhoods and bounds checks

### NADI-006 — Tejas scalar field + `Patch` model

**Description:**
The light itself (ARCHITECTURE §World model): a scalar field over the grid, each cell holding `0..FOOD_MAX` units of light energy, and the `Patch{center, radius, intensity, lifetime}` value that shapes it. Cells under a patch are what feed producers from v1.2. (`nadiloka/tejas.py`.)

**What needs to be done:**
- The field: per-cell energy in `0..FOOD_MAX`; `FOOD_MAX` comes from the world config, alongside the other Tejas patch parameters (target count, radius, intensity, lifetime).
- `Patch{center, radius, intensity, lifetime}` as a plain data model — lifecycle behaviour belongs to NADI-007.
- Illumination: applying a patch writes intensity into the cells within its radius (via NADI-005's neighbourhood helpers), **clamped so no cell ever exceeds `FOOD_MAX`**; a patch near the border illuminates the clipped disc.
- A field read/decrement API — v1.2's harvest debits the field through it; keep the hot path cheap (pure arithmetic, no allocation churn).
- Unit tests: the `FOOD_MAX` clamp holds under overlapping patches; edge-clipped illumination is correct.

**Dependencies:** NADI-005

**Expected result:**
A Tejas field over the grid that patches illuminate correctly, with every cell clamped to `0..FOOD_MAX`.

**Acceptance criteria:**
- [ ] Per-cell field in `0..FOOD_MAX`; `FOOD_MAX` and patch parameters from the world config
- [ ] `Patch{center, radius, intensity, lifetime}` model defined
- [ ] Illumination writes intensity over the patch radius, clipped at grid edges
- [ ] No cell exceeds `FOOD_MAX`, including under overlapping patches (unit test)
- [ ] Field read/decrement API in place for v1.2's harvest

### NADI-007 — Patch lifecycle in the update-Tejas phase

**Description:**
The living part of the light: the update-Tejas phase (the first hook of `Meru.step`, stubbed in NADI-003) now drives the patch lifecycle — **ignite → illuminate → fade → disappear → respawn** — holding a target number of live patches. Placement is seeded through the World's single RNG, so the light dance replays exactly.

**What needs to be done:**
- Fill the update-Tejas hook: each tick, advance every live patch (age it, decay its intensity over the lifetime), remove expired patches, and re-ignite new ones to hold the configured target live-patch count.
- Ignite: choose seeded random cells via the World's one `random.Random`, passed explicitly — never module-level `random.*` (RNG discipline, ARCHITECTURE §Key invariants and pitfalls).
- Fade: intensity decays over the patch lifetime and the field under it dims accordingly; a fully faded patch disappears and frees its cells.
- Respawn elsewhere: replacement patches ignite at fresh seeded positions, so the light visibly moves across the grid.
- All lifecycle parameters (target count, radius, intensity, lifetime) come from the world config (ARCHITECTURE §Configuration and parameters).
- Unit tests: the ignite/decay/respawn transitions fire correctly; the live-patch count holds at the target; same seed → identical patch placements and lifecycle timing.

**Dependencies:** NADI-006, NADI-003

**Expected result:**
Ticking the world makes patches appear, live their lifetime, fade, and reappear elsewhere — deterministically under a fixed seed.

**Acceptance criteria:**
- [ ] The update-Tejas phase advances, expires, and re-ignites patches each tick
- [ ] The live-patch count holds at the configured target
- [ ] Patch placement uses the World's single seeded RNG, passed explicitly
- [ ] Unit test: ignite → fade → disappear → respawn transitions fire correctly
- [ ] Unit test: same seed → identical patch placements and lifecycle over N ticks

### NADI-008 — Field-energy counter and energy-band test

**Description:**
The measurable half of the phase (ARCHITECTURE §Counters and measurement): the total-field-energy counter joins the v0 counter set (tick, total Tejas energy, live patches), and the **energy-band test** encodes the v0.2 DoD — the field fluctuates around a steady intended level, with no runaway growth or collapse.

**What needs to be done:**
- A total-field-energy counter (sum over all cells) and a live-patch counter, emitted per tick on the structured counter line from NADI-004.
- Wire both into the demo runner's output; they become the renderer header content in v0.3.
- The band test: over M ticks with a fixed seed at the reference parameters, total field energy stays within a configured band — no runaway growth, no collapse to zero.
- Assert alongside it that no cell exceeds `FOOD_MAX` at any sampled tick.
- Keep the band, M, and the reference parameters in config so v0.3's smoke run (NADI-011) reuses the same guard.

**Dependencies:** NADI-007

**Expected result:**
Counter logs show total energy fluctuating around the intended level; the band test pins it under a fixed seed.

**Acceptance criteria:**
- [ ] Total field energy and live-patch count emitted per tick on the counter line
- [ ] Unit test: total energy stays within the configured band over M ticks (fixed seed)
- [ ] Unit test: no cell exceeds `FOOD_MAX` over the run
- [ ] The determinism harness still passes: same seed → byte-identical counter logs with the new counters included
- [ ] Band, M, and reference parameters live in config, reusable by the v0.3 smoke run
