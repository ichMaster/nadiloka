## v1.1 — Digitant and the Jati loader

### NADI-016 — Digitant dataclass + two-clock skeleton

**Description:**
The Digitant — the individual creature — as a dataclass carrying the state fields from ARCHITECTURE §Digitant, plus the **two-clock skeleton**: a cheap deterministic **body-tick** hook driven from the act phase (NADI-003) and the **brain** hook it hands over to. The body acts, then the brain re-evaluates — that order is fixed here and never changes. (`nadiloka/digitant.py`.)

**What needs to be done:**
- `digitant.py`: `Digitant{id, pos, needs, health, age, lifespan, repro_cd, jati}` plus a `genome` placeholder — equal to the species defaults until v6 (ARCHITECTURE §Genome and heredity); the body-tick reads traits from the individual, not the descriptor.
- `needs` per ARCHITECTURE §Digitant: hunger, energy, social, each clamped to a range (the rates arrive in v1.2).
- The body-tick hook: pure arithmetic, no LLM (the standing ROADMAP invariant) — aging (`age += 1`) now, with stubbed slots for movement/feeding/metabolism that later phases fill.
- The brain hook: called after the body-tick; a no-op until NADI-017 wires the pysm machine in.
- Wire the act phase (NADI-003's hook) to run body-tick then brain hook per living digitant.
- Ids allocated by the World (a monotonic counter); any randomness comes only from the World's seeded `random.Random`, passed explicitly — module-level `random.*` is forbidden (ARCHITECTURE §Key invariants and pitfalls, RNG discipline).

**Dependencies:** NADI-003 (v0.1 — Meru phase order)

**Expected result:**
A Digitant can be constructed, ticked, and aged deterministically; the act phase runs body then brain, in that order, for every living digitant.

**Acceptance criteria:**
- [ ] `Digitant` carries `id, pos, needs, health, age, lifespan, repro_cd, jati` + a `genome` placeholder (species defaults)
- [ ] Body-tick is pure arithmetic (aging now; movement/feeding/metabolism slots stubbed); no module-level `random.*`
- [ ] The act phase runs body-tick then the brain hook per living digitant, in that order (unit test on call order)
- [ ] Unit test: ticking ages a digitant deterministically — same seed yields an identical age/state trace
- [ ] Ids are World-allocated and monotonic; construction touches no global state

### NADI-017 — Minimal pysm brain honouring the invariants

**Description:**
The tiny pysm mode machine behind the Digitant's brain hook — **one mode** for the stationary producer, but built to the pysm invariants from the start (ARCHITECTURE §Key invariants and pitfalls) so every later species (Chara in v4, Vyaghra in v5) inherits a correct base. Per the two-clock (ARCHITECTURE §Two-clock), the brain never acts: it only re-evaluates the mode from fresh needs; the body acts in the current mode.

**What needs to be done:**
- A pysm state machine with a single mode (Idle) attached to the Digitant's brain hook; construction selected by the Jati `brain` field as a named engine preset (ARCHITECTURE §Extension points — yamlfsm is not a dependency yet).
- **Manual initial `on_enter`:** pysm does not fire the initial state's `on_enter` itself — call it explicitly after `initialize()`.
- **Internal transitions** for any action without a mode change, so within-mode re-evaluation never re-fires `on_enter` (the seam v1.2's Idle/Hungry re-evaluation plugs into).
- **Event data via `cargo`** (kwargs on Event), never `input=` — pysm's `input` is for transition matching and silently drops the event on mismatch.
- A pinning test for each pitfall, per ARCHITECTURE §Testing and CI (the pysm pitfalls each get a pinning test).
- Keep the brain side-effect-free over world state: it reads needs and sets the mode, nothing else.

**Dependencies:** NADI-016

**Expected result:**
Every Digitant carries a live, correctly initialized pysm brain; the pitfall behaviours are pinned by tests before any real behaviour is built on them.

**Acceptance criteria:**
- [ ] A one-mode pysm brain wires into the Digitant brain hook, selected via the Jati `brain` preset name
- [ ] Pinning test: the initial `on_enter` fires — called manually after `initialize()`
- [ ] Pinning test: within-mode re-evaluation is an internal transition; `on_enter` does not re-fire
- [ ] Pinning test: event data travels via `cargo`; an `input=`-style mismatch silently drops the event (documented and pinned)
- [ ] The brain only sets the mode; it never mutates world state

### NADI-018 — Jati loader + `species/bindu.yaml`

**Description:**
The declarative species layer: `jati.py` loads and validates a YAML descriptor, and the first descriptor — `species/bindu.yaml` — is authored. Everything per-species lives in the descriptor, never in code (ARCHITECTURE §Configuration and parameters); from v4 on, adding a trophic level is writing YAML, and this loader is what makes that true. (`nadiloka/jati.py`, `species/bindu.yaml`.)

**What needs to be done:**
- `jati.py`: load a `species/*.yaml` file into a validated Jati object with the fields from ARCHITECTURE §Species descriptive language: `jati`, `ahara`, `harvest` (`gain`, `radius`), `yields`, `needs` (values + rates), `brain` (a named pysm preset), `reproduce` (adulthood, energy threshold, cooldown, cap, cost), `move`, `signals`, `life` (lifespan, health rules).
- Validation: required fields present, numeric ranges sane (non-negative gains/rates/costs), `move` limited to the known styles, unknown `brain` presets rejected — with clear errors naming the file and field.
- Author `species/bindu.yaml`: `ahara: tejas`, `move: stay`, empty `signals`, the one-mode brain preset, and harvest/needs/reproduce/life numbers as the v1 reference parameters (tuned against the field in v1.2–v1.3).
- Digitant instantiation from a Jati: species defaults populate the digitant's fields; the `genome` placeholder equals the defaults (ARCHITECTURE §Genome and heredity).
- The loader is a pure descriptor-to-object step — no engine state touched; `pyyaml` is already a dependency (NADI-001).

**Dependencies:** NADI-001 (v0.1 — skeleton & tooling)

**Expected result:**
`bindu.yaml` loads into a validated Jati; malformed descriptors fail loudly and precisely; a digitant instantiated from it carries the species defaults.

**Acceptance criteria:**
- [ ] `jati.py` loads and validates all descriptor fields (`jati, ahara, harvest, yields, needs, brain, reproduce, move, signals, life`)
- [ ] `species/bindu.yaml` authored: `ahara: tejas`, `move: stay`, a one-mode brain, v1 reference numbers
- [ ] Unit tests: malformed descriptors (missing field, bad range, unknown move style, unknown brain preset) each raise a clear validation error
- [ ] A digitant instantiated from the Jati carries the species defaults (genome placeholder included)
- [ ] No species parameter is hardcoded in engine code

### NADI-019 — Population wiring: spawn, spatial hash, snapshot iteration

**Description:**
Puts the population dict and the v0 spatial grid to real use: spawn N Bindu from `bindu.yaml` at seeded positions, fill the spatial hash from live digitants each tick, and make the act phase iterate a **snapshot** of the population — the invariant that keeps mid-tick births from breaking iteration (ARCHITECTURE §Key invariants and pitfalls, §One tick cycle phase 3). Delivers the v1.1 DoD.

**What needs to be done:**
- Spawn: the World spawns N Bindu from the loaded Jati at positions drawn from the seeded World RNG (config-driven N; free cells only, on the bounded grid).
- The rebuild-grid phase (NADI-005) fills the spatial hash from live digitants every tick; neighbour queries return exactly the in-range agents.
- The act phase iterates a **snapshot list** of the population: a newborn injected mid-act does not act this tick and acts from the next — the pinning test v1.3's reproduction path (NADI-023) relies on.
- Renderer: digitants render as glyphs over the Tejas field; the counters header gains per-species population (the v1 counter set begins here — ARCHITECTURE §Counters and measurement).
- Determinism: two same-seed runs produce identical spawn positions and counter logs (the standing harness from ARCHITECTURE §Testing and CI).

**Dependencies:** NADI-016, NADI-018, NADI-005 (v0.2 — grid + neighbourhood helpers)

**Expected result:**
From `bindu.yaml` the world spawns N Bindu at seeded positions; they appear in the renderer and the spatial index; ticking ages them deterministically — the v1.1 DoD.

**Acceptance criteria:**
- [ ] From `bindu.yaml` the world spawns N Bindu at seeded positions (config-driven, free cells only)
- [ ] The rebuild-grid phase fills the spatial hash from live digitants; unit test: neighbour lookup returns exactly the in-range agents
- [ ] Unit test: a newborn injected mid-act does not act this tick (snapshot iteration), and acts the next tick
- [ ] Digitants appear in the ASCII renderer; per-species population joins the counters header
- [ ] Determinism: two same-seed runs produce identical spawn positions and counter logs
