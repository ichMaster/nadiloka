## v4.2 — Needs, brain, and seek-prey

### NADI-039 — The full brain: Wander / Hungry / Sleeping

**Description:**
The full consumer brain: three pysm modes with **internal transitions**, driven by needs (hunger, energy) whose rates come from Jati. The pysm invariants from ARCHITECTURE §Key invariants and pitfalls hold from the start: manual initial `on_enter`, internal transitions (no `on_enter` re-fire), event data via `cargo` — each pinned by a test. (`nadiloka/digitant.py`.)

**What needs to be done:**
- Extend the pysm brain (NADI-017) into a named preset with three modes — Wander / Hungry / Sleeping — selected by the Jati `brain` field (ARCHITECTURE §Extension points: named presets until yamlfsm).
- Needs (hunger, energy) with per-tick rates read from the Jati `needs` field; the body-tick decays them, then the brain re-evaluates the mode from fresh needs (the two-clock: the body acts in the current mode, the brain picks the next).
- Threshold transitions from Jati (never code constants): hunger above threshold → Hungry; fed with low energy → Sleeping (recover); otherwise Wander.
- Re-evaluating into the same mode is an **internal transition** — no `on_enter` re-fire (the pysm pitfall, pinned by a test).
- Manual initial `on_enter` after `initialize()`; any event data rides in `cargo`, never `input=`.
- Keep the brain tiny: mode re-evaluation only; every action stays in the cheap, deterministic body-tick.

**Dependencies:** NADI-017

**Expected result:**
A Chara's mode tracks its needs deterministically: it wanders when fed, goes Hungry past the threshold, sleeps to recover energy.

**Acceptance criteria:**
- [ ] Wander / Hungry / Sleeping implemented as pysm modes behind a named `brain` preset
- [ ] Needs rates and mode thresholds come from the Jati descriptor, not code (ARCHITECTURE §Configuration and parameters)
- [ ] Unit test: modes transition on need thresholds (hunger crossing → Hungry; recovery → back to Wander)
- [ ] Unit test: staying in a mode is an internal transition — `on_enter` does not re-fire
- [ ] Unit test: the initial mode's `on_enter` fires exactly once (manually, after `initialize()`)

### NADI-040 — `seek_prey` movement

**Description:**
`move: seek_prey` — when Hungry, step toward the **nearest known prey** via the spatial hash; wander otherwise. This is the directed movement that keeps the food chain alive: under random movement predator and prey rarely meet (ROADMAP §Notes on the ladder). Builds on the v2 movement styles. (The body-tick movement path.)

**What needs to be done:**
- Add `seek_prey` to the `move` styles (`stay`/`wander`/`seek_light` from v2): when the brain is Hungry, query the spatial hash for the nearest live prey of the `ahara` species and step toward it — bounded grid, no wrap, occupancy respected as in the wander step (NADI-026).
- Deterministic nearest-prey choice: distance first, ties broken by **agent id (id-sorted)** — never hash/dict order (ARCHITECTURE §Key invariants and pitfalls); pinned by a test.
- Outside Hungry (Wander / Sleeping), fall back to the mode's movement: the wander step, or staying put.
- No prey within perception range → wander; the spatial-hash query bounds the cost — no whole-population scan, the body-tick stays cheap.
- Movement remains a body-tick action driven by the current mode; the brain (NADI-039) only picks the mode (two-clock).
- Wire the style through the Jati `move` field so `chara.yaml` selects it declaratively; the seeded RNG is used for any residual random choice (RNG discipline).

**Dependencies:** NADI-039, NADI-028

**Expected result:**
Hungry Chara visibly converge on Bindu; fed Chara drift. Same seed → same paths.

**Acceptance criteria:**
- [ ] `move: seek_prey` steps toward the nearest live prey when Hungry; wanders otherwise
- [ ] Unit test: nearest-prey choice is deterministic — equidistant prey resolve by the id-sorted tie-break
- [ ] Unit test: no prey in range → a wander step (in bounds, occupancy respected)
- [ ] Unit test: two same-seed runs produce identical movement paths
- [ ] Prey lookup goes through the spatial hash (no whole-population scan)

### NADI-041 — Chara reproduction and death

**Description:**
The v1.3 lifecycle reused for Chara, configured entirely by `chara.yaml`'s `reproduce`/`life` fields — **descriptor-only, no engine change**. If an engine edit turns out to be needed, that is an abstraction leak to fix generically (ROADMAP §Notes on the ladder). (`species/chara.yaml`; the reproduction and death paths.)

**What needs to be done:**
- Fill/confirm the `chara.yaml` `reproduce` fields (adulthood, energy threshold, cooldown, cap, cost) and `life` fields (lifespan, health rules).
- Verify the v1.3 reproduction path (NADI-023) drives Chara unchanged: adulthood + energy threshold + cooldown, under `MAX_POP` → spawn in a nearby free cell, debit `cost`, newborn added after the snapshot (acts from the next tick).
- Verify the death phase (NADI-024) removes starved (zero-health) and old (past `lifespan`) Chara — coexisting with the `eaten` removals from NADI-037 in the same phase.
- Health dynamics per the v1.2 semantics: sustained starvation lowers health, feeding recovers it — parameters from `life`, not code.
- Split the births/deaths/population counters per species (Bindu vs Chara) so v4.3's time-series has its inputs (ARCHITECTURE §Counters and measurement).
- Hand-run the v4.2 DoD end to end: wander fed, seek hungry, reproduce rich, die starved or old.

**Dependencies:** NADI-038, NADI-023, NADI-024

**Expected result:**
Chara reproduce when rich and die when starved or old — through exactly the v1.3 code paths, configured by the descriptor alone.

**Acceptance criteria:**
- [ ] Unit test: Chara reproduction honours the Jati threshold/cooldown/cap and debits `cost`
- [ ] Unit test: the lifecycle is descriptor-only — a synthetic species YAML with different `reproduce`/`life` values follows them with no engine change
- [ ] Unit test: death removes exactly the starved and over-lifespan Chara; `eaten` removals coexist in the same phase
- [ ] Newborn Chara do not act on their birth tick (snapshot iteration)
- [ ] Per-species births/deaths/population counters emitted per tick
