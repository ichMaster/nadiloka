## v4.3 — Two-level balance

### NADI-042 — `demos/chara.py` two-species scenario

**Description:**
The runnable two-species scenario — Bindu + Chara on Tejas — the stage every balance experiment in this phase runs on. Patterned on `demos/bindu.py` (v1.3), with the scenario's world config and an explicit seed. (`demos/chara.py`.)

**What needs to be done:**
- `demos/chara.py`: wire World + Tejas + both species (`species/bindu.yaml`, `species/chara.yaml`) + the renderer over the standard demo loop.
- A world config for the scenario (dataclass or file, per ARCHITECTURE §Configuration and parameters): grid size, seed, Tejas patch parameters, `MAX_POP`, initial population per species, demo tick count — every number in the config, none in code.
- `--seed` on the CLI; the seed is always explicit (config or CLI), never implicit (RNG discipline).
- The renderer distinguishes the two species (glyphs) and shows per-species population and per-level energy counters in the header; the web viewer rides the same additive snapshot unchanged.
- A short fixed-seed run wired into CI as a cheap sanity check (the long smoke guard is NADI-044).

**Dependencies:** NADI-040, NADI-041

**Expected result:**
`python demos/chara.py --seed 42` shows Bindu colonies blooming on the light and Chara hunting them, with both levels' counters live in the header.

**Acceptance criteria:**
- [ ] `demos/chara.py` runs both species over the standard loop and exits cleanly after N ticks
- [ ] All scenario numbers live in the world config + species YAML; `--seed` accepted on the CLI
- [ ] Determinism harness: two same-seed runs produce identical counter logs
- [ ] The renderer shows both species distinctly plus per-species population and per-level energy
- [ ] A short fixed-seed CI run completes without error

### NADI-043 — Time-series counters + phase-offset indicator

**Description:**
The measurable half of the Lotka-Volterra target: per-level population time-series (CSV/structured log, ARCHITECTURE §Counters and measurement) and an **out-of-phase oscillation indicator** — the number the v4.3 DoD is asserted against. Strictly observation: counters read the world, never mutate it. (Counters + `demos/chara.py` wiring.)

**What needs to be done:**
- Per-level population (and energy) time-series emitted per tick as structured log lines / CSV — consumable by tests and offline plotting.
- A phase-offset indicator computed over a sliding window of the two population series (e.g. the lag of the peak cross-correlation, or peak-offset detection) — cheap enough to run inside the demo loop.
- The indicator reports a verdict — **out of phase** vs in phase vs no signal — with documented thresholds; the window and thresholds live in config, not code.
- Guard against false positives: constant (dead) or near-flat series must report "no signal", never "out of phase".
- Wire the indicator into the demo header and persist it in the counters log so NADI-044's smoke test can assert on it.
- Unit tests on synthetic series: shifted sine-like series are detected as out of phase; identical/in-phase series are not; flat series yield "no signal".

**Dependencies:** NADI-042

**Expected result:**
A `chara.py` run leaves a CSV of both population series and a phase-offset verdict a test can assert on.

**Acceptance criteria:**
- [ ] Per-level population time-series written as CSV/structured log each tick
- [ ] The phase-offset indicator reports lag/offset and a verdict over a configurable window
- [ ] Unit test: synthetic out-of-phase series detected; in-phase and flat series are not (no false positives)
- [ ] The indicator shows in the demo header and lands in the persisted counter log
- [ ] Read-only invariant: same-seed runs with the indicator on/off produce identical world state and counter logs

### NADI-044 — Balance tuning + the Lotka-Volterra smoke guard

**Description:**
Tune the two-level parameters for coexistence — in the world config and species YAML **only** (ARCHITECTURE §Configuration and parameters: nothing tunable lives as an engine constant) — then freeze them as the regression reference and encode the v4 readiness criterion as a long fixed-seed smoke test. This is where ROADMAP §Notes on the ladder expects the pain: balance is hardest at v4–v5. (World config, `species/*.yaml`, the smoke suite.)

**What needs to be done:**
- Tune for coexistence: Tejas patch supply, Bindu harvest/reproduce numbers, Chara `gain`/`yields`, needs rates, reproduce cost — touching only the world config and `species/*.yaml`, never engine code.
- Iterate against the NADI-043 time-series and indicator, looking for the classic signature: a Bindu peak, a lagging Chara peak, a Bindu trough, a lagging Chara trough.
- The long fixed-seed smoke test: K ticks at the tuned reference parameters; assert **both populations stay above zero** for the whole run and the phase-offset indicator detects **out-of-phase oscillation**.
- Runaway/collapse guard: saturation at `MAX_POP` or the extinction of either level fails the test — flagged as a regression on the tuned parameters (ROADMAP §v4.3 Tests).
- Commit the tuned numbers as the reference config: the regression baseline every later change (v5 builds directly on it) is judged against.
- Briefly document the tuning rationale next to the reference config — which knobs mattered and in which direction — so v5's three-level tuning starts informed.

**Dependencies:** NADI-043

**Expected result:**
v4 readiness (Layer 5): Bindu and Chara oscillate out of phase and neither level dies out over the long run; CI holds that line from now on.

**Acceptance criteria:**
- [ ] Tuning confined to the world config + species YAML (no engine-code constants changed)
- [ ] Smoke test: a long fixed-seed run keeps both populations above zero for the full K ticks
- [ ] Smoke test: the phase-offset indicator detects out-of-phase oscillation over the run
- [ ] Explosion (`MAX_POP` saturation) or extinction of either level fails the guard
- [ ] The tuned parameters are committed as the reference config — the regression baseline
- [ ] The smoke test runs in CI (pure Python, no network, per ARCHITECTURE §Testing and CI)
